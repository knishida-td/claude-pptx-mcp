# 日本市場向けPPTX生成ルール

このファイルは日本語プレゼンテーション生成時に適用する追加ルールを定義する。

---

## フォント

日本語プレゼンでは以下のフォントを使用する：

```javascript
const FONT = "Hiragino Kaku Gothic Pro W3";
```

- macOSに標準搭載。Windows向けの場合は「游ゴシック」等に変更
- **チャート（addChart）はデフォルトでArialになる。** 以下のプロパティを必ず指定する：
  ```javascript
  catAxisLabelFontFace: FONT, catAxisLabelFontSize: 7,
  valAxisLabelFontFace: FONT, valAxisLabelFontSize: 7,
  dataLabelFontFace: FONT, dataLabelFontSize: 8,
  legendFontFace: FONT, legendFontSize: 7,
  ```
  フォント名だけでなくサイズも指定する。デフォルトのサイズは大きすぎてスライドの他テキストと不釣り合いになる。
- `typeface`, `panose`, `pitchFamily`, `charset`をOOXMLテンプレートで指定する場合：
  ```xml
  <a:latin typeface="Hiragino Kaku Gothic Pro W3" panose="020B0300000000000000" pitchFamily="34" charset="-128"/>
  <a:ea typeface="Hiragino Kaku Gothic Pro W3" panose="020B0300000000000000" pitchFamily="34" charset="-128"/>
  ```

## 写真素材

### 日本市場向け → 日本人モデルを使う

- **ぱくたそ（PAKUTASO）** https://www.pakutaso.com/ を第一候補にする
  - 日本人モデルの写真が豊富、商用利用無料
  - CDN URL: `https://user0514.cdnw.net/shared/img/thumb/{filename}.jpg`
  - 検索: `https://www.pakutaso.com/search.html?offset=0&limit=50&search={URLエンコード済みキーワード}`
- Pexels/Unsplashは「Asian」で検索しても日本人に見えないモデルが多い。日本市場向けなら避ける
- ぱくたそでも外国人モデル写真があるため（ファイル名に`modelf`等）、必ずダウンロード後に目視確認する

### 商品スライド → 実際の商品写真を使う

美容家電・デバイス等の商品紹介スライドでは：
- **実際に販売されている商品のプロダクト写真**を使う（ブランド公式サイト、ECサイト等）
- 汎用ストックフォト（サロン施術風景等）は使わない
- テキストオーバーレイのない、商品単体の写真を優先する
- Shopifyの場合: `/products.json` APIで商品画像URLを一括取得できる

### 画像の入れ方（最重要）

**画像は積極的に使いたい。ただし「画像前提でレイアウトしたスライド」に入れる。**
画像を前提としていないスライド（テーブル、チャート、番号リスト等）の余白に無理やり小さく入れるとダサくなる。

```
✅ 画像が活きるスライド（レイアウト段階から画像の場所を確保する）:
- 商品・ブランド紹介スライド → 右半分を画像エリアにする
- Before/After → 視覚的に対比を示す
- 顧客ジャーニー → 右側に商品写真を配置
- 3カラムのカードヘッダーにアイコンや写真

❌ 画像を無理に入れてダサくなるスライド:
- テーブルの横の余白に小さい商品画像
- チャートの隅に関係ない画像
- 投資計画・ロードマップ・推進体制の脇
```

**ルール**: 画像を入れたいなら、レイアウトの時点でそのスライドを「画像あり前提」で設計する。
テキストやデータ中心のスライドに後から画像を足すのはNG。

## 画像のアスペクト比とクロップ

### PptxGenJSの`sizing`プロパティは使わない

`sizing: "cover"` や `sizing: "contain"` はPowerPointで正しく描画されない場合がある。

### 正しい方法：PILで事前クロップ

```python
from PIL import Image

def smart_crop(src, target_ratio_w, target_ratio_h, out_path, gravity="center"):
    img = Image.open(src)
    w, h = img.size
    target_r = target_ratio_w / target_ratio_h
    current_r = w / h

    if current_r > target_r:
        # 横長すぎ → 左右をクロップ
        new_w = int(h * target_r)
        left = (w - new_w) // 2
        img = img.crop((left, 0, left + new_w, h))
    else:
        # 縦長すぎ → 上下をクロップ
        new_h = int(w / target_r)
        if gravity == "top":
            top = 0
        elif gravity == "center":
            top = (h - new_h) // 2
        else:
            top = h - new_h
        img = img.crop((0, top, w, top + new_h))

    out_w = 1200
    out_h = int(out_w / target_r)
    img = img.resize((out_w, out_h), Image.LANCZOS)
    img.save(out_path, quality=90)
```

### ルール

1. スライド上の画像サイズ（w, h）からアスペクト比を計算
2. PILで元画像をその比率にクロップ（被写体の重要部分を確認して`gravity`調整）
3. PptxGenJSではクロップ済み画像を`w`, `h`指定で挿入（`sizing`プロパティなし）
4. **正方形に近い商品写真**は横長スライド枠に無理に入れない。画像枠自体を正方形寄りに変更する

### 同じ画像を別サイズで再利用しない

同じ画像をスライドごとに異なるw:hで配置すると縦横比が崩れる。**使用箇所ごとにアスペクト比が異なる場合は、別ファイルとしてクロップし直す。**

```
例:
- P1: cover.jpg (10:5.625 = 16:9) → フルブリード
- P20: cover_wide.jpg (10:2.5 = 4:1) → 上部バナー用に別途クロップ
```

### クロップ後の目視確認必須

- 被写体の重要部分（顔、商品本体）が切れていないか確認
- 人物写真は顔全体（額〜顎）が入っているか確認
- 商品写真は全体が見えているか確認

## レイアウトルール

### 縦中央寄せ

コンテンツブロックは、タイトル領域とフッター（キーメッセージ or ページ番号）の間で**縦中央**に配置する。

```
計算方法:
- タイトル領域の下端: 約 y=0.85
- キーメッセージ上端: y=4.65（ある場合）
- ページ番号上端: y=5.25（キーメッセージなしの場合）
- 利用可能高さ = 下端 - 0.85
- コンテンツ開始y = 0.85 + (利用可能高さ - コンテンツ高さ) / 2
```

### 左右中央揃え

テーブルやコンテンツブロックはスライドの左右中央に配置する。

```
計算方法:
- x = (10 - w) / 2
- 例: w=8.6 → x = (10 - 8.6) / 2 = 0.7
```

**テーブルの場合、`colW`の合計が`w`と一致しないと見た目が中央にならない。**
- `colW`の合計 < `w` → 右側に空白ができ、テーブル全体が左寄りに見える
- **必ず `colW` の合計 = `w` をコード上で検算してからビルドする**
- 5列以上のテーブルで特にミスしやすい。電卓で足し算確認を怠らない

**テーブル以外の左右中央も同様に検算する。**
- 番号付きリスト・ステップ系レイアウトで、最左要素のxと最右要素の(x+w)を確認
- ブロック全体の幅 = 最右端 - 最左端
- 開始x = (10 - ブロック幅) / 2
- **x座標を固定値で決め打ちしない。必ずブロック幅から計算する**

```javascript
// NG: colW合計 6.95 ≠ w 8.6 → 右に1.65"の空白
s.addTable(data, { x: 0.7, w: 8.6, colW: [1.2, 0.75, 0.65, 1.3, 1.55, 1.5] });

// OK: colW合計 8.6 = w 8.6
s.addTable(data, { x: 0.7, w: 8.6, colW: [1.5, 0.95, 0.8, 1.6, 1.9, 1.85] });
```

テーブル以外（シェイプ、テキストボックス等）も同様に `x = (10 - w) / 2` で中央配置する。

### テキスト溢れ防止

- 数値+単位（例: `¥16,000 / 60%`）はフォントサイズ14pt以下で収める
- ボックス幅に対して文字列がはみ出さないか、画像QAで必ず確認
- KPI数値のフォントサイズは内容に応じて調整（5文字以内なら36pt、それ以上は段階的に縮小）

### 要素間の重なり防止

- 表の最終行とその下のセクション（バーチャート等）が重ならないよう、y座標を必ず計算確認
- `要素の終了y = 開始y + 高さ` を計算し、次の要素の開始yがそれ以上であることを確認
- **コンテンツ高さの合計が利用可能スペースを超えないことを必ず検算する**
  - `totalContentH = 要素1高さ + gap + 要素2高さ + ...`
  - `totalContentH > availableH` の場合、要素の高さ・フォントサイズ・間隔を縮小して収める
  - テーブルの`rowH`配列の合計と、計算に使う`tableH`変数が一致しているか確認（例: `rowH: [0.4, 0.4, 0.4, 0.4]` → `tableH = 1.6` であり `1.8` ではない）
  - **テーブル直後に要素を配置する場合**: `次の要素のY = テーブルY + rowH合計 + gap(最低0.15)`。テーブルの下端を必ず計算してからY座標を決める。固定値で期待効果バー等を置かない
  - **テーブルの列幅は内容に合わせる**: 長いテキストが入る列の幅が狭くて改行される一方、短いテキストの列に余白がある場合は、列幅を再配分する。「全列均等」や「コード上のキリの良い数字」ではなく、内容量に応じた幅にする
  - **オブジェクト被りチェックは絶対ルール**: コードを書いた後、全要素のY座標と高さを計算し、次の要素との間にgap(最低0.15)があることを必ず検算する。これはQAの前にコード上で実施する。被りが見つかったら修正してから生成する
  - **PowerPointでのレンダリングはLibreOfficeより要素が大きくなる傾向がある**: テーブルのrowHや要素間のgapは余裕を持たせる（最低0.15、できれば0.2）

### ダークスライドでのテキスト視認性

- 暗い背景（navy等）上にボックスを配置する場合、**ボックスを不透明な白（`fill: { color: "FFFFFF" }`）、テキストをネイビー等の濃い色**にする
- `transparency` 付きの半透明白ボックス + 白文字はコントラスト不足になりやすい。避ける
- ボックス内テキストが折り返さないよう、ボックス幅を十分に確保する（日本語は1文字あたり約0.18〜0.2インチで概算）

## ファイル出力

### デスクトップに保存しない

```javascript
// NG
const outputPath = "/Users/.../Desktop/output.pptx";

// OK
const outputPath = "/tmp/pptx-project/output.pptx";
```

プロジェクトディレクトリまたは`/tmp`に出力し、`open`コマンドで開く。

### PowerPointでの再読み込み

同名ファイルを再度開く場合、先にアクティブなプレゼンを閉じる：

```bash
osascript -e 'tell application "Microsoft PowerPoint" to close active presentation saving no' 2>/dev/null
sleep 0.5
open "/tmp/pptx-project/output.pptx"
```

## 提案資料の内容ルール

### 提案資料の標準構成（20枚以上）

薄い提案書は信頼されない。**提案資料は20枚以上**で厚みを持たせる。以下の構成を標準テンプレートとする。

```
推奨構成（24枚の実績テンプレート: /tmp/sample-ec-proposal-v5.js）

[導入]
 1. タイトル（Type A）— 社名御中 + 提案書タイトル + サブタイトル + 日付
 2. アジェンダ — 番号付きリスト + ページ参照

[理解・分析 6-8枚]
 3. 貴社の理解 — 会社概要 + 商品写真（実写）
 4. ブランドポジション — SWOT 2×2グリッド
 5. 市場環境 — 大見出し数字（3列）
 6. 市場トレンド — 3カラムグリッド
 7. 競合分析 — テーブル
 8. 現状分析 — チャート（実データ）+ 課題リスト
 9. チャネル別 — ドーナツチャート + チャネル解説
10. 課題の構造 — 因果フロー

[施策提案 8-10枚]
11. セクション扉（Type B）
12-19. 各施策 × 2枚ずつ（概要 + 詳細/Before-After/KPI）

[効果・実行 4枚]
20. 投資計画 — テーブル
21. 期待効果 — 折れ線チャート + KPIテーブル
22. ロードマップ — ガントチャート
23. 推進体制 — 2×2カード

[クロージング]
24. ネクストステップ（Type D）
```

**レイアウトの多様性**: 連続するスライドで同じパターンを使わない。定義ブロック→チャート→テーブル→3カラム→Before/After のように変化をつける。

### 二重リサーチの実行（必須）

提案資料を作成する前に、**一次リサーチ**と**ファクトチェックリサーチ**の2段階を必ず実行する。

#### 一次リサーチ（市場・競合・企業の全体像）
- 対象企業の直近6ヶ月のプレスリリース（PR TIMES等）を確認
- アンバサダー起用、新商品発売、チャネル展開、提携・M&Aなどの最新施策を把握
- 競合の最新動向（売上、新施策、チャネル変更）も同様にリサーチ

#### ファクトチェックリサーチ（公式サイトで商品・サービスの実在確認）
一次リサーチの後、**対象企業の公式サイト・ECサイトを直接確認**して以下をファクトチェックする：
- **商品ラインナップ**: 公式ストアの商品一覧ページをWebFetchで確認。存在する商品・廃盤商品を正確に把握
- **価格**: 公式サイトまたはAmazon/楽天の現在の販売価格を確認
- **販売チャネル**: 実際にどこで買えるかを確認（公式EC、Amazon、楽天、コンビニ、ドラッグストア等）
- **廃盤情報**: 「販売終了のお知らせ」等の公式告知を確認

**ファクトチェックなしに資料に書いてはいけない事項:**
- 「○○商品がない」→ 公式サイトで確認したか？
- 「○○商品がある」→ 廃盤になっていないか？
- 「○○チャネルに未参入」→ 実は既に販売していないか？

#### 典型的な失敗例
- ソイプロテインが既に販売されているのに「ソイプロテイン商品なし」と記載
- 廃盤済み商品（EAA BEAUTY等）を現行商品として提案に組み込む
- RTDプロテインドリンクが既にあるのに「RTD未参入」と記載

#### ルール
- **既に実行済みの施策を「提案」として書かない**
- **既に存在する商品を「ない」と書かない**
- **廃盤商品を「ある」と書かない**
- リサーチ結果を踏まえ、「既にやったこと」と「まだやっていないこと」を明確に区分してから提案を構成する

### 各提案に期待効果（KPI）を付ける

提案スライドには必ず具体的な期待効果を数字で示す。クライアントが投資判断できるレベルの粒度にする。

```
例:
- 「女性新規顧客 +2万人/年」「コンビニRTD売上 3〜5億円/年」
- 「ARPU 8,200円→15,800円（1.9倍）」「定期購入率 25%→40%」
- 「粗利率 +3〜5pt改善」「原料コスト変動リスク ▲30%」
```

「○○を強化する」のような定性的な表現だけでは不十分。必ず数値目標を付ける。

### ビジネスモデルの構造変数への対処を含める

新規施策だけでなく、既存事業のビジネスモデルにおける重要変数（LTV、定期購入率、クロスセル率、解約率等）への対処を必ず含める。

- EC/D2C事業なら: 定期購入率、継続率、客単価、クロスセル導線、LINE/SNS活用
- GYM/店舗事業なら: ARPU、プラン付帯率、退会率、クロスセル
- 新規獲得のROIは既存顧客LTV改善の1/5程度。LTV改善は最もROIが高い施策として常に検討する

## QA チェックリスト（日本語PPTX追加項目）

生成後、標準QA（pptx.md参照）に加えて以下を確認：

- [ ] フォントがHiragino Kaku Gothic Pro W3になっているか
- [ ] 写真の人物が日本人に見えるか（日本市場向けの場合）
- [ ] 商品スライドに実際の商品写真を使っているか
- [ ] 画像のアスペクト比が崩れていないか（被写体が歪んでいないか）
- [ ] 画像クロップで被写体の重要部分が切れていないか
- [ ] コンテンツが縦中央に配置されているか（上寄り/下寄りでないか）
- [ ] コンテンツが左右中央に配置されているか（テーブルのcolW合計 = w か）
- [ ] Type Dスライド（CTA/エンド）のコンテンツが縦横ともに中央か（x座標を固定値で決め打ちしていないか）
- [ ] 番号付きリスト等のブロック要素で、最左端〜最右端の幅を計算し x=(10-幅)/2 で開始しているか
- [ ] テキストがボックスからはみ出していないか
- [ ] 要素同士が重なっていないか
- [ ] コンテンツ高さの合計が利用可能スペース内に収まっているか（totalContentH ≤ availH）
- [ ] テーブルのrowH合計と計算用tableH変数が一致しているか
- [ ] ダークスライドのボックス内テキストが十分なコントラストで読めるか
- [ ] 同じ画像を異なるアスペクト比で使い回していないか（別途クロップが必要）
- [ ] チャートのフォントがHiragino/指定フォントになっているか（デフォルトはArial）
- [ ] チャートのフォントサイズが他テキストと釣り合っているか（7〜8pt推奨）
- [ ] 各提案に具体的な期待効果（KPI数値）が付いているか
- [ ] 既存事業のLTV改善施策が含まれているか（新規施策だけになっていないか）
- [ ] 二重リサーチを実施したか（一次リサーチ + 公式サイトでのファクトチェック）
- [ ] 「○○がない」と書いた箇所を公式サイトで確認したか（実は存在する可能性）
- [ ] 廃盤商品を現行商品として記載していないか
- [ ] 全要素のY座標+高さを計算し、次の要素とのgap≥0.15を検算したか（オブジェクト被り防止の絶対ルール）
- [ ] テーブル内のテキストが改行されていないか（改行される場合は列幅を再配分）
- [ ] P14のように要素が多いスライドは情報を減らして整理したか
- [ ] 出力先がデスクトップでないか
